# Permission And Guardrails V1

## Purpose

This dossier defines the shared kernel contract for permission, approval, and guardrail outcomes.

BreadBoard currently has several related but distinct control systems:

- permission broker decisions
- guardrail counters and structured guardrail events
- policy-pack denials and allowlists
- completion guards and plan-mode guards
- host-visible approval projections

A multi-engine BreadBoard needs a shared semantic split between:

- the request for permission or policy evaluation
- the decision that was returned
- the audit trail generated by that decision

---

## Contract role

This contract family sits at the boundary where a kernel must decide whether a model-suggested action is:

- allowed automatically
- denied automatically
- escalated for host approval
- transformed or constrained by guardrails

The contract must remain usable across:

- Python reference engine
- future TS engine
- replay/conformance fixtures
- host projections such as TUI, sidebar, and web app

---

## Shared semantics that must be frozen

A conforming engine must preserve the semantics of:

- permission request identity
- permission request category / pattern / metadata
- decision outcome (`allow`, `deny`, `ask`, `approve-once`, `approve-always`, etc.)
- decision reason / source (policy pack, config allowlist, interactive approval, auto-allow)
- guardrail record identity and type
- model-visible denial / validation consequence where applicable
- host-visible audit record for approvals and denials

The exact host projection can vary. The kernel meaning cannot.

---

## Important distinction

Permission and guardrails are related, but not identical.

### Permission

Permission is the question:

- may this tool call or action proceed?

### Guardrails

Guardrails are broader:

- they may block, downgrade, warn, inject context, or annotate outcomes
- they may operate before tool execution, after tool execution, or at turn-completion time

The contract family should allow those systems to coexist without collapsing them into one undifferentiated event type.

---

## Non-goals

This contract does not freeze:

- exact UI copy for approval prompts
- exact TUI event grouping
- particular policy-pack authoring syntax
- filesystem layout of stored permission rules
- Python exception types used internally

Those may be implementation-specific.

---

## Recommended first schema content

A first-pass `bb.permission.v1` should capture:

- `request_id`
- `category`
- `pattern`
- `scope`
- `metadata`
- `decision`
- `decision_source`
- `reason`
- `requires_host_interaction`
- `audit_refs`

Guardrail-specific records may later deserve their own contract family, but for now this dossier should at least define the semantic relationship.

## Why this matters for TypeScript

A TS engine can only be trusted if it preserves denial/approval semantics exactly enough that host-facing behavior and replay outcomes remain honest.

This is especially important for E4 / harness-calculus work, where approval and guardrail behavior is often a defining harness trait.
