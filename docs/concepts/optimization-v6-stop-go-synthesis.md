# Optimization V6 Stop/Go Synthesis

This note closes the scoped V6 program.

It is based on the live V6 platform state:

- the public optimize kernel was frozen
- the Codex dossier ↔ OpenCode `1.2.17` pair ran as bounded live cohort cells
- the harder replay/config follow-on stayed within the same artifact set
- the live-result boundary was made explicit
- the private search layer was hardened only for repeated live-cell pressure

## What the live studies taught

The current platform can already express the next important class of optimization work without new public optimize artifacts:

- Nano-only bounded live transfer cells
- Nano-first follow-ons with audited Mini escalation
- transfer-supported claims inside named cohorts
- explicit experiment-only vs durable-change boundaries
- bounded private stopping and blockedness handling

That means the main V6 question was answered in the affirmative:

> the existing V5 public kernel was sufficient for real live cohort-cell use.

## Repeated-shape pressure check

The central V6 question for future planning is whether repeated live use forced a missing public shape.

Current answer:

- no new public artifact was forced
- no package-graph or study-manager pressure was justified
- no reward/search public ontology was justified
- no new backend family was justified

The repeated-shape pressure that did appear was private:

- dead Nano sweeps need bounded early stop
- blocked semantic dominance needs bounded freeze
- Mini audits that change nothing need bounded freeze

Those were solved inside the private search layer and did not justify a new public optimize artifact.

## No-V7 criteria

The default conclusion after V6 should now be:

> **use the platform**

Do **not** do `V7` by default if all of the following remain true:

- live cohort cells run without schema pain
- no repeated public-artifact gap appears across multiple cells
- claim tiers remain honest under repeated use
- Mini remains rare and auditable
- the most valuable next work is selecting better cells and running more studies

That is the current state.

## What would justify V7

Only justify `V7` if:

- at least two live studies expose the same missing public shape
- private search duplication becomes a real maintenance burden
- or claim-tier/generalization reporting repeatedly breaks down in a way that cannot be solved with doctrine or documentation alone

Anything weaker than that should not justify another kernel-planning round.

## DARWIN handoff signals

Future work should move toward DARWIN, not V7 optimize, if pressure becomes:

- many-cohort orchestration
- adaptive study scheduling
- persistent cross-run archive state
- diversity or novelty search
- async candidate fleets
- merge, lineage, or skill-management behavior beyond bounded cohort promotion

That work is outer-loop research orchestration, not bounded optimize-kernel work.

## Recommendation

Current recommendation:

- keep using the current optimize platform
- run more live cohort cells under the existing doctrine
- keep reward/search private
- keep Nano-first economics intact
- only reopen optimizer-kernel planning if repeated live evidence clearly forces it

That is a clean V6 stopping point.
