# Optimization V5 Transfer Cohorts and Generalization Evidence

Optimization V5 starts where V4 stops.

V4 made mixed-evidence package lanes real:

- `TransferSliceManifest`
- two live 3-family package lanes
- slice-aware promotion/generalization evidence
- Nano-first / auditable Mini-escalation doctrine
- backend-private mixed-evidence package search policy

What V4 did **not** solve is the next pressure:

> how to make honest cross-package transfer/generalization claims without inventing a package graph, a study-runtime ontology, or a public reward/search-policy layer.

## Public center

V5 keeps the public center narrow.

It adds one new public non-kernel artifact:

- `TransferCohortManifest`

This artifact exists because `TransferSliceManifest` is atomic, while V5 needs one bounded grouping
layer for claims like:

- this package pair
- this shared model-tier policy
- this bounded family emphasis

It is **not**:

- a package graph
- a package-set ontology
- a study manager
- a campaign manifest
- a search policy
- a reward surface

## Value seam

V5 makes the optimize ↔ reward boundary explicit.

The doctrine is:

- evaluation is truth
- objective suites are derived and inspectable
- reward-like ranking stays private search or telemetry
- no public reward artifact appears
- no public search-policy artifact appears

This avoids the repo drifting into two incompatible value languages.

## Claim tiers

V5 introduces explicit claim tiers in promotion and cohort-aware benchmark evidence:

- `package_local`
- `transfer_supported`
- `cohort_supported`

The first V5 tranche only proves `package_local` and `transfer_supported`.

That is intentional. It keeps claims honest.

## First proving ground

The first live V5 cohort is:

- Codex dossier current package
- OpenCode `1.2.17`

with a shared bounded emphasis on:

- tool-guidance clarity
- bounded edit / support honesty

This is the right first proving ground because:

- both packages are already live in V4
- it has high methodology value
- it is still Nano-valid
- it avoids widening to 4-family local packages before transfer/generalization is real

## Cost policy

V5 keeps the same model doctrine:

- `GPT-5.4 Nano` is the default
- `GPT-5.4 Mini` is explicit escalation only

In the first cohort tranche:

- the baseline cohort study is Nano-only
- Mini is documented but disabled for the first proof
- fairness and stopping rules are explicit in suite metadata and cohort docs

## What stays private

The following stay backend-private:

- scalarization
- reward-like ranking
- candidate scheduling
- early stopping heuristics
- Mini escalation governance
- cohort-aware penalties and pruning

The public center remains:

- evaluation suites
- objective suites
- transfer slices
- transfer cohorts
- benchmark results
- promotion evidence

## What V5 still does not do

V5 does **not** introduce:

- DARWIN campaign state
- archive / island / QD / genealogy ontology
- package graphs
- experiment-study manifests
- public reward suites
- public search-policy manifests
- 4+ family widening as the center
- online self-tuning

The point of V5 is to make transfer/generalization claims honest, not to reopen optimizer architecture wholesale.
