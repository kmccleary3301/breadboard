# BreadBoard direction assessment - distilled findings (for adversarial review)

## Thesis under review
A maximalist engine for agent-harness design: build any modern harness (research/engineering/other) and clone other harnesses as config (E4), with the cleanest minimal set of expressive primitives. Priorities (stated order): maintainability; atomicity/composability; generalizability; development velocity; versatility; dev-x; interpretability; intuitive configuration; forward compatibility; minimal blast radius against existing BreadBoard core behavior and existing E4 gains (lower priority than the rest)

## Orchestrator's personal inspection (ground truth, verified by direct reads)
- bb.agent_config_surface.v2.schema.json: 51KB, 20 closed top-level sections, only 3 shared $defs; additionalProperties:false; feature-union shape.
- agent_configs/templates/minimal_harness.v2.yaml: 66 lines, fully commented, boots mock harness offline; genuine bright spot.
- config/e4_lanes/*.yaml: 22 lanes, 13.7k lines total; accepted lanes embed machine-refreshed pins (hashes/timestamps/artifact roles). Source-of-truth is hand-tracked YAML validated by scripts/e4_parity/lane_definitions.py; refresh_lane_descriptor_pins.py rewrites literal pins.
- scaffold_e4_target_lane.py: new-lane input is small (lane_id/config_id/target family+version), BUT emits a per-lane Python builder skeleton and comparator skeleton -> capture bottoms out in target-specific code (registered adapter or per-lane builder), not pure config.
- agentic_coder_prototype/api/e4/router.py: read-only evidence browser (schemas/lanes/claims/catalog/registries/ledger/records/coverage) + POST reverify. Strong integrity checks (catalog binding drift 409s). No execution/authoring surface.
- compilation/helper_runtime_primitives.py: hand-rolled _require_text/_require_bool/... validation duplicating what JSON Schema declares; 14 schema-version constants; bundle compilers finalize records.

## Subsystem auditor verdicts (independent GPT-5.6 readers, file-grounded)

### schemas
VERDICT: The schema program is the right strategic direction: machine-readable contracts, explicit runtime truth, composable E4 evidence stages, registries, and lifecycle state are strong foundations for config-defined harness cloning. Its current shape does not meet the thesis’s “cleanest minimal set.” It is a governed catalog of everything BreadBoard has needed, with a capable primitive core buried inside. The highest-value correction is to name that core, make config sections and provider behaviors registered compositions, converge duplicated common types, and demote derived, campaign, and superseded records from the published language. With an executable end-to-end tutorial and bundle-level validation, newcomers could author against it. Today they can inspect records, but they still need repository lore to assemble a harness correctly.

- [high/maintainability / minimality / atomicity-composability] The published “minimal expressive harness primitive language” is an inventory of 61 active kernel schemas plus 13 active E4 schemas, not a minimal basis. Several entries are campaign evidence or specialized host records rather than irreducible harness concepts. The suite reads as successful accretion organized after the fact: work/task variants, five terminal record types, multiple coordination reference slices, config explanation/mutation/effective graph, and numerous narrowly scoped execution records all sit in the kernel pack. (files: contracts/kernel/schemas/README.md, contracts/kernel/schemas/bb.work_item.v1.schema.json)
- [high/generalizability / maintainability / forward compatibility] `bb.agent_config_surface.v2` is a monolithic captured-product union rather than a small compositional language. Its 1,883 lines hard-code Anthropic prompt caching and rate limits, RLM scheduling, todo behavior, LSP integration, patch splitting, specific guardrails, long-run review, team bus policy, and multiple alternate spellings/shapes (`id`/`team_id`, edge `from`/`source`, `to`/`target`, `mode`/`modes`). Adding a new provider or harness behavior tends to grow this central schema and its blast radius. (files: contracts/kernel/schemas/bb.agent_config_surface.v2.schema.json, contracts/kernel/registries/config_surface_fields.v1.json)
- [high/atomicity-composability / interpretability / maintainability] Task and coordination concepts overlap without a clear canonical composition. `work_item`, `task`, and `distributed_task_descriptor` each model task identity/state/delegation; coordination pack/slice/reference-slice families then embed or reproduce signals, directives, verdicts, jobs, and events. `bb.coordination_slice.v2` says it supersedes five reference-slice schemas, yet those five remain listed in the active kernel pack rather than `legacy_frozen`. (files: contracts/kernel/schemas/bb.work_item.v1.schema.json, contracts/kernel/schemas/bb.task.v1.schema.json)
- [high/maintainability / interpretability / forward compatibility] Common types are only partly shared. Current runtime v2 schemas use `bb.kernel.common.v1`, but active v1 schemas locally copy visibility triples, actor references, artifact references, and SHA-256 regexes. Some hash regexes accept uppercase hex while the canonical definition requires lowercase. `effective_config_graph`, `projection_event`, and `work_item` each define their own visibility dialect instead of the canonical visibility object. (files: contracts/kernel/schemas/bb.kernel.common.v1.schema.json, contracts/kernel/schemas/bb.effective_config_graph.v1.schema.json)

### regen_machinery
VERDICT: The subsystem is the right architectural direction but not yet the right final shape for the thesis. Its evidence products and convergence design are serious: isolated capture, scoped claims, stable catalog hashing, explicit write ownership, merged ledger rows, and two-pass fixed-point checks directly address parity credibility. The limiting fact is that the generic DAG is partly representational: normalize/replay/compare do not execute as composed stages, new accepted lanes are not automatically captured, and Oh-My-Pi parity depends on a large target-specific Python compiler. A lane matching an existing stored-package template can be close to zero-Python; an arbitrary new harness cannot. Preserve the evidence DAG, but make stage execution and lane discovery descriptor-driven and demote target-specific Python to raw probing only.

- [high/atomicity/composability, generalizability] The advertised capture→normalize→replay→compare pipeline is not an executable generic pipeline. In `run_lane.py`, normalize, replay, and compare return `skipped: true`; compare merely resolves a registry entry, and the claim/reverify command performs the real rerun/diff. The factoring is useful as an evidence vocabulary, but the runtime seam is effectively capture→lane-specific C4 command, so new replay engines cannot be composed from descriptor data alone. (files: scripts/e4_parity/run_lane.py, scripts/e4_parity/adapters/semantic_replay.py)
- [high/generalizability, versatility] The registry does not make arbitrary new harness capture zero-Python. It dynamically imports only capture adapters in `run_lane.py`; registered translators and comparators are not executed there. A harness fitting the existing stored-package/accepted-artifact templates may be added without Python, but any novel capture protocol or semantic projection requires a new adapter implementation. Oh-My-Pi proves the cost: its adapter directly imports multiple target-specific projection tables and embeds target paths/assertions/claim construction. (files: scripts/e4_parity/run_lane.py, contracts/kernel/registries/e4_adapters.v1.json)
- [high/development velocity, intuitive configuration, forward compatibility] Lane #23 is not a reliably config-only marginal addition. Reverify stages are generated from descriptors, but capture stages in the canonical DAG are explicitly enumerated for current P3/P5/P6 lanes (with only the special north-star group selected by comparator ID). A new accepted lane can be reverifed yet never have its capture artifacts regenerated unless `regenerate_evidence.py` is edited or it happens to join that special group. It also needs coordinated inventory artifact roles/report-role mirror/comparator lane membership and often ledger behavior seed data. (files: scripts/e4_parity/regenerate_evidence.py, scripts/e4_parity/seed_atomic_feature_ledger.py)
- [high/maintainability, generalizability, minimal primitives] The Oh-My-Pi compiler adapter is a target-specific subsystem disguised behind one registry entry: it dispatches a global Python `PROJECTIONS` table, has an accepted-lane allowlist, hard-coded source/canonical paths and timestamps, special P3.1 versus remaining-record branches, manually constructs capture/replay/comparator/parity/claim/manifest artifacts, and embeds exact comparator assertions. This is valuable migration code but not evidence that the core abstraction can clone arbitrary harnesses as pure config. (files: scripts/e4_parity/adapters/oh_my_pi_compiler_capture.py)

### lane_configs
VERDICT: The lane engine is directionally right for the thesis: its stage vocabulary, scoped claims, adapters, projections, and semantic assertions can describe varied harness evidence without changing core runtime behavior. The checked directory is not the right user-facing boundary. Compact descriptors are barely hand-authorable; the 7,292-line P6.6 file is unequivocally a machine-pinned packet masquerading as config. Preserve the expressive compiled representation, but treat it as generated truth. The product surface should be a small intent manifest compiled into a lock/evidence packet, with presets, profiles, explain/diff tooling, and a strict source/generated split. Agent-config v2 has a sound conceptual spine, but captured dossiers and disabled subsystem defaults should be referenced or compiled, not authored inline.

- [high/maintainability; intuitive configuration; development velocity; interpretability] The P6.6 “lane config” is a generated evidence archive, not a hand-authorable descriptor: 7,292 lines embed raw-capture manifests, literal target observations, six full work-item records repeated in replay outputs, comparator results, support claims, evidence manifests, node-gate output, a 98-row atomic-feature ledger, hashes, byte counts, and large substitution path tables. Declarative intent is only the small outer lane shell (identity, adapter/projection choice, required record IDs, claim scope, CT); almost everything under `normalize.config.packet_constants.payload_templates` and `substitutions` is embedded evidence or generated binding machinery. (files: config/e4_lanes/oh_my_pi_p6_6_task_job_subagent.yaml)
- [high/atomicity/composability; maintainability; minimal blast radius] Evidence is duplicated across configuration and canonical artifact files. The P6.6 descriptor copies the contents of `raw_capture_manifest.json`, work items, replay/comparator reports, support claim, evidence manifest, node gate, and ledger rows even though `roles` already names those files. This creates coupled hash refreshes and makes a small evidence update touch a huge user-facing file. (files: config/e4_lanes/oh_my_pi_p6_6_task_job_subagent.yaml)
- [high/development velocity; dev-x; intuitive configuration] The surface conflates four ownership domains: author intent, execution recipe, provenance/freeze identity, and promotion evidence. Even the relatively readable Pi lane requires authors to supply archive hashes/bytes, freeze hash fragments, reverify CLI details, run IDs, source-path inventories, and duplicated acceptance/comparator assertions. (files: config/e4_lanes/pi_p5_l1_cli_config_context_tool_surface.yaml)
- [high/interpretability; maintainability; forward compatibility] Lane size and representation vary by orders of magnitude (roughly 84–7,292 lines), and JSON-formatted documents use `.yaml`. A user cannot infer whether a lane is source, generated lockfile, dossier, or evidence packet from directory or extension. (files: config/e4_lanes/oh_my_pi_p6_6_task_job_subagent.yaml, config/e4_lanes/oh_my_pi_p3_1_effective_config_graph_compiler.yaml)

### api_surface
VERDICT: The subsystem is a solid operational inspection API for E4 evidence, with good namespace isolation, artifact integrity checks, and a real OpenAPI drift gate. Its current shape does not carry the product thesis end to end. The central nouns are campaign artifacts, and the only E4 write operation reverifies a claim. A harness designer still needs repository files, scripts, and environment flags to author lanes, start capture, regenerate outputs, and understand effective behavior. The highest-value correction is to retain these diagnostic reads while adding explicit harness, lane, capture-job, and explanation resources, then generate the public TS client from the versioned OpenAPI contract. That would turn a useful internal observability surface into the engine API the thesis calls for.

- [high/versatility / dev-x / development velocity] The E4 surface is chiefly an internal evidence browser, not yet a product API for designing harnesses. Thirteen of its fourteen domain routes are GETs; the sole mutation reverifies an existing claim. Clients cannot create or update a lane, submit a dossier/config, trigger capture or regeneration, inspect a capture job, or ask why an effective harness behaves as it does. This leaves the thesis workflow outside the API and forces users back to repository files and scripts. (files: agentic_coder_prototype/api/e4/router.py, agentic_coder_prototype/api/e4/models.py)
- [high/dev-x / maintainability / forward compatibility] The public TypeScript client does not expose E4 at all. It is hand-written around legacy-style unversioned `/sessions`, `/models`, and related paths, while the server defaults to removing those aliases. A default `createBreadboardClient` can therefore target routes that default `create_app` returns 404 for. There is no shared generation or compile-time check tying these client methods and DTOs to the exported OpenAPI contract. (files: sdk/ts/src/client.ts, sdk/ts/src/types.ts)
- [high/maintainability / forward compatibility / interpretability] Python E4 response DTOs are permissive projections (`extra="allow"`) and duplicate only selected fields from authoritative JSON Schemas. OpenAPI is generated from those Pydantic classes, while kernel TypeScript types are generated independently from JSON Schema. CI can prove that checked-in OpenAPI matches Python, and that generated kernel TS matches schemas, but it does not prove semantic equivalence across the Python projection DTOs, OpenAPI components, and kernel TS types. Fields can disappear from the documented HTTP shape or change optionality without a cross-contract failure. (files: agentic_coder_prototype/api/e4/models.py, agentic_coder_prototype/api/e4/router.py)
- [high/atomicity/composability / intuitive configuration / versatility] The API's capture-related runtime emission is controlled through process environment flags and session creation side effects rather than an explicit capture resource. Callers cannot request a capture mode, discover its dialect or output location through a stable DTO, monitor emission, or retry it independently of creating a runtime session. (files: agentic_coder_prototype/api/cli_bridge/runtime_emission.py, agentic_coder_prototype/api/cli_bridge/app.py)

### compilation_primitives
VERDICT: The direction is right at the contract boundary, but the subsystem is not yet the minimal expressive heart claimed by the thesis. `finalize_record()` plus schema contracts and the provenance graph are strong foundations. Around them, captured-harness projection conveniences, duplicated validation, heuristic tool semantics, hard-coded merging, and dual authority switches have accumulated into a grab-bag. The remedy is consolidation rather than more wrappers: schemas/lifecycle as generated truth, one versioned config compilation context, atomic compilers separated from E4 assemblers, and explicit hook/middleware and permission-lattice primitives. Until then BreadBoard can describe many harness artifacts, but cannot reliably compose arbitrary harness behavior from a clean minimal algebra.

- [high/atomicity/composability; generalizability] The layer is only partly orthogonal. `finalize_record()` is a true primitive, but `helper_runtime_primitives.py` is a projection-oriented grab-bag: some functions construct rich semantics and hard-code policy (`compile_resource_access_bundle` fixes workspace scope, snapshot immutability, model visibility, content-addressed storage, approval identities, and default redaction refs), while other functions merely bundle unrelated pass-through finalizations. Bundling resource/ref/blob and protocol/route/policy creates coupling that prevents independent composition and substitutes captured-lane convenience for minimal harness primitives. (files: agentic_coder_prototype/compilation/helper_runtime_primitives.py)
- [high/maintainability; forward compatibility] Contract source-of-truth is duplicated in Python. Allowed source kinds, capability types, visibility values, discovery/exposure modes, effect statuses, and resource operations are copied from JSON Schemas into module-level sets and checked manually before the same record is schema-validated. The capability schema already contains values absent from `_ALLOWED_CAPABILITY_TYPES` (for example `slash_command` and `extension_module`), demonstrating drift. Visibility enums are copied again into three tables. (files: agentic_coder_prototype/compilation/helper_runtime_primitives.py, agentic_coder_prototype/compilation/visibility_adapters.py)
- [high/maintainability; development velocity; intuitive configuration] The v2 loader is schema-driven at entry, then re-validates a large and growing subset by hand. `_validate_v2()` repeats required fields and hundreds of lines of RLM/long-running type, range, and enum checks already suitable for JSON Schema. It also coerces with `int()`/`float()`, so strings may pass semantic validation despite schema types, and future schema changes require synchronized Python edits. This is a major config-front-door maintenance hazard. (files: agentic_coder_prototype/compilation/v2_loader.py, contracts/kernel/schemas/bb.agent_config_surface.v2.schema.json)
- [high/forward compatibility; interpretability; intuitive configuration] Schema-version dispatch does not form a robust extensible version registry. Any absent discriminator is called v1, only exactly v2 is recognized, and `_agent_config_surface_validator()` silently selects the v1 schema for every non-v2 argument. Meanwhile `version: 2` without the v2 discriminator takes a special legacy path. Two version concepts (`schema_version` and numeric `version`) therefore encode overlapping authority and permit surprising treatment of malformed or future documents. (files: agentic_coder_prototype/compilation/v2_loader.py, contracts/kernel/schemas/bb.agent_config_surface.v1.schema.json)

### sdk_ts
VERDICT: The schema-consumer foundation is the right shape for the thesis: generation is real, exhaustive by pack, runtime-validated, and kernel-core mostly composes rather than duplicates generated contracts. The public product around it is not yet the right shape. Thirteen kernel-stack packages without a workspace, release story, or umbrella harness API turn good internal seams into consumer friction; `@breadboard/sdk` compounds this by naming an unrelated CLI client as the obvious entrypoint. Preserve the schema generator, contract package, runtime/driver boundary, and host adapters. Consolidate publication units, add one workspace gate and one harness entrypoint, and make host-facing semantic adapters explicit before presenting this as an external SDK.

- [high/dev-x; development velocity; interpretability] There is no coherent external harness-builder entrypoint. `@breadboard/sdk` sounds like the umbrella package but is only the legacy CLI-bridge HTTP/SSE client and has no dependencies on contracts, core, workspace, or backbone. A new consumer must discover and assemble several separately named experimental packages, with no top-level installation recipe or end-to-end harness example. (files: sdk/ts/package.json, sdk/ts/README.md)
- [high/maintainability; development velocity; atomicity/composability] The 13-package kernel stack is premature at its present distribution maturity. Backbone directly depends on seven sibling packages, host bridges on nine, and each package repeats build/install orchestration. The seams between kernel contracts, generic driver API, concrete drivers, and host-specific bridges are defensible; separate workspace, backbone, host-kits, AI transport, T3, Temporal, and three driver distributions are not yet supported by an actual workspace/release surface. This maximizes package graph and versioning burden before a coherent consumer product exists. (files: sdk/ts-backbone/package.json, sdk/ts-host-kits/package.json)
- [high/development velocity; maintainability; dev-x] Package-local test scripts use `npm install ../...` and long `npm --prefix` chains as dependency orchestration. With no root workspace manifest, verification and local development are order-dependent, slow, mutate package installs/locks, and provide no single generated-contract/typecheck/test gate for external contributors. (files: sdk/ts-backbone/package.json, sdk/ts-kernel-core/package.json)

### evidence_conformance
VERDICT: The subsystem is directionally right for the thesis: exact-scope claims, frozen upstreams, replay normalization, semantic comparators, explicit exclusions, and hash-linked provenance are a credible foundation for “capture a harness.” The central correction is to reserve byte identity for reproducible generation, not define it as behavioral parity. Semantic equivalence must be the conformance bar, with role-specific ordering and volatility rules. The evidence DAG should also become cheaper to carry: one canonical generated index, content-addressed bulky payloads outside normal Git review, stable evidence roots, single-source CT descriptors, and an automated target-version compatibility/bump workflow. Without those changes, every new harness release turns a strong proof model into a maintenance campaign.

- [high/generalizability / forward compatibility] Byte identity is used at two different conceptual levels without sufficiently sharp terminology. It is the right bar for deterministic regeneration of generated accepted artifacts, but not for harness conformance. `docs/conformance/E4_COOKBOOK_V2.md` requires migrated artifact roles to be byte-identical and the watch set to reach a byte-identical fixed point. Meanwhile the actual behavioral proof is semantic in `docs/conformance/e4_target_support/oh_my_pi_p6_0_l5_memory_compaction/comparator_report.json`. Treating the former as a parity bar overconstrains harmless serialization changes: object-key order, ordering of set-like observations, path formatting, and metadata. The surveyed packet embeds fixed `generated_at_utc` and `run_id` values throughout `raw_capture_manifest.json`, `bb_replay_result.json`, `comparator_report.json`, and the support claim; real recapture timestamps or new run IDs necessarily break hashes even when behavior is equivalent. (files: docs/conformance/E4_COOKBOOK_V2.md, docs/conformance/e4_target_support/oh_my_pi_p6_0_l5_memory_compaction/raw_capture_manifest.json)
- [high/maintainability / development velocity] The tracked evidence surface is already too large and duplicative for routine review. `docs/conformance/ct_scenarios_v1.json` is 6,395 lines and `docs/conformance/e4_artifact_catalog.json` is 4,496 lines; the catalog reports a 124,871-byte ledger, an 89,337-byte kernel-event JSONL payload, and a 67,354-byte freeze manifest. Four near-duplicate catalog forms (`e4_artifact_catalog.json`, `e4_artifact_catalog_full_snapshot.json`, `e4_artifact_catalog_post_report_snapshot.json`, `e4_artifact_catalog_stable_snapshot.json`) amplify diff noise. Lane directories also retain raw stdout plus parsed probe output, replay, comparator, parity wrapper, scans, and manifests. Generated evidence is sustainable only if CI regenerates it and reviewers inspect compact semantic deltas; the current checked-in duplication makes ordinary changes high-churn and obscures meaningful evidence changes. (files: docs/conformance/ct_scenarios_v1.json, docs/conformance/e4_artifact_catalog.json)
- [high/forward compatibility / development velocity] A new upstream harness version causes a broad recapture cascade rather than a localized compatibility judgment. The old claim remains valid only for its pinned version, but advancing support requires a new/updated freeze row, evidence captures, replay, comparator hashes, support claim, evidence manifest, catalog, inventory/CT bindings, snapshots, and fixed-point regeneration. `docs/conformance/E4_TARGET_VERSIONING.md` documents the manual eight-step update, while `docs/conformance/E4_COOKBOOK_V2.md` requires agreement across lane definition, registry, inventory, claim, CT row, score binding, and accepted artifacts. This is safe but expensive, and it does not define a compatibility window or a first-class result such as unchanged / semantically compatible / breaking. Even a timestamp-only or ordering-only target-output change can force the same machinery as a behavioral break. (files: docs/conformance/E4_TARGET_VERSIONING.md, docs/conformance/E4_COOKBOOK_V2.md)

### tests_gates
VERDICT: The test architecture supports the thesis in substance: it treats schemas, lanes, evidence lineage, determinism, and regeneration safety as first-class contracts. The strongest tests pin real behavior, especially loader closure, schema validity, write containment, freshness, and fixed-point convergence. The architecture falls short operationally because CI does not run the 678-test E4/compilation battery, the public E4 router has no visible endpoint tests, and golden byte truth is interleaved with semantic contracts. Make the battery an enforced, timed, sharded gate; expose a genuinely fast contract lane; and automate schema-change impact/regeneration. Those changes would preserve rigorous capture parity while reducing the cost of evolving the minimal primitive set.

- [high/maintainability; development velocity; minimal blast radius] The main E4 battery is not wired into CI. A repository-wide workflow search found no invocation of `tests/e4_parity` or `tests/compilation`; `.github/workflows/ci.yml` directly runs only `tests/test_e4_target_freeze_manifest.py` for the E4 freeze job. The 678-test claim therefore describes a local battery, not an enforced merge gate. Regressions in lane validation, schema lifecycle, compiler projections, regeneration fixed-point behavior, or loader closure can merge unless caught indirectly by another job. (files: .github/workflows/ci.yml, tests/e4_parity/test_regen_front_door.py)
- [high/atomicity/composability; development velocity; forward compatibility] Changing one closed schema field has a wide manual update surface. A field rename/addition can require the schema, minimal example, loader fixtures, exact report-shape assertions, generated artifacts, catalogs and their hashes, support-claim bindings, readiness reports, and fixed-point outputs to move together. Concrete coupling appears in the exact key/check lists in `test_validate_lane.py`, root closure and example mapping in `test_runtime_v2_schemas.py`, catalog byte/hash checks in `test_build_artifact_catalog.py`, and transitive freshness checks in `test_e4_report_hash_freshness.py`. This is high-cost when the change is intentional, even though the failure fan-out is useful for accidental drift. (files: tests/e4_parity/test_validate_lane.py, tests/e4_parity/test_runtime_v2_schemas.py)
- [high/maintainability; versatility; dev-x; interpretability] Router endpoints are effectively untested. `agentic_coder_prototype/api/e4/router.py` defines health, schema, lane, claim, catalog, registry, ledger, records, reverify, and coverage routes, but a search across `tests/` found no E4 router/TestClient coverage. Missing cases include filtering/pagination, 404s, malformed cached documents, reverify 409 behavior, path/schema lookup, response-model conformance, and app prefix integration. This leaves the public inspection/reverification surface outside the otherwise strict contract net. (files: agentic_coder_prototype/api/e4/router.py, agentic_coder_prototype/api/e4/models.py)

### vision_docs
VERDICT: The direction was right: the project found a credible kernel-edge vocabulary, enforced honest pinned-version claims, wired runtime records, and replaced much bespoke lane machinery with registered data-driven paths. The subsystem does not yet prove the owner’s full thesis. It proves a narrower and valuable result: several harness surfaces can be captured and replayed through a general evidence engine with controlled extensions. “Pure config” now includes adapter and packet-builder code, and “done” is defined per campaign rather than for the universal harness engine. The next step should be reconciliation, not another campaign: publish the thesis, map the original primitives to current families, measure target-specific code, and require an unseen-harness capture through existing extension points.

- [high/generalizability] The repository has campaign completion criteria, but no durable definition of done for the full owner thesis. Phase 17 proves selected demonstrations; Phase 18 and Phase 19 then declare narrower campaigns complete. None establishes that the primitive language can build “ANY modern harness,” defines a representative harness capability basis, or requires an unseen-harness sufficiency test. The public index does not state the maximalist/minimal-language thesis at all. Current state therefore meets the North Star campaign’s operational proof, not the universal harness-engine thesis. (files: docs_tmp/phase_17/BB_NS_MASTER_PLAN.md, docs_tmp/phase_17/BB_NS_FINAL_REPORT.md)
- [high/atomicity/composability] “Capture a pokemon as pure config” was silently weakened into “config/data plus registered code adapters.” Phase 17 says capture means YAML plus evidence and names the comparator registry as the only code extension point. Phase 18 allows adapter capture and only requires dispositions for lanes that remain bespoke. Phase 19 accepts a generic compiler adapter with a central projection table, then ADR-AV-5 permits whole-packet construction with lane `records` and `packet_constants`. This removes per-lane builder filenames, but substantial capture semantics still live in Python. Calling that pure config overstates the achieved abstraction. (files: docs_tmp/phase_17/BB_NS_MASTER_PLAN.md, docs_tmp/phase_18/BB_CL_MASTER_PLAN.md)
- [high/maintainability] The original “accepted set is fixed” of 14 primitives became a much larger contract estate without a documented lineage table showing which Phase 15 primitive remained canonical, split, merged, became evidence-only, or was superseded. Phase 18 introduced pack classification and Phase 19 lifecycle classification, which help mechanically, but neither is a semantic reconciliation of the original language. A maintainer cannot tell from the durable docs whether the minimal primitive thesis still means the Phase 15 set, the kernel pack, or all active-production schemas. (files: docs_tmp/phase_15/BB_E4_PRIMITIVE_PARITY_MASTER_PLAN.md, docs_tmp/phase_15/BB_E4_PRIMITIVE_DECISION_LEDGER.json)
- [high/development velocity] The durable docs treat the project thesis as archaeology. `docs/INDEX.md` explicitly places phase plans under local `docs_tmp/`, but provides no stable vision/architecture document that states the harness-engine thesis, primitive-language boundary, E4 claim grammar, and current limitations. New contributors can learn how to run or author artifacts, but cannot discover what architectural result the system is optimizing for. (files: breadboard_repo_integration_main_20260326/docs/INDEX.md, docs_tmp/phase_15/BB_E4_PRIMITIVE_PARITY_MASTER_PLAN.md)
